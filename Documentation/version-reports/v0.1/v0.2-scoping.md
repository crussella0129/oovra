# Oovra v0.2 Scoping Report

A snapshot analysis of v0.1's feature surface, with concrete proposals for v0.2. The guiding principle: **don't add features that are shortcuts for what already works.** Every proposed feature is checked against the question "can this be replicated with existing operators?" — features that pass that check are documented as **discovered features** rather than additions; features that fail (or fail painfully) are documented as **genuine additions**.

The user's primary focus is improving `compare`. Part 1 of this report covers that in detail. Parts 2–4 cover the rest of the v0.2 surface.

This document is itself a v0.2 planning artifact — it should be revisited when v0.2 work begins, used as the scope contract, and eventually archived once v0.2 ships.

---

## Part 1 — Improvements to `compare` (Primary Focus)

### Current state of `compare` (v0.1)

| Dispatch path | Currently does | Currently misses |
|---|---|---|
| **Atomic vs atomic** (same order, both atomic) | Line-level body diff via `similar::TextDiff::from_lines`; tracks changes to 3 header fields (`name`, `version`, `meta`) | Changes to `id`, plus all 4 conditional fields if present (none should be on atomics, but the validator's job is to catch that — diff stays silent) |
| **Composed vs composed** (same order, both composed) | Set-based diff over `composed_of` using a `HashMap` keyed by id — reports `added`, `removed`, `version_changed` | **Position-of-input changes (reorderings)**, **the body itself**, **all 8 other header fields** (`name`, `id`, `version`, `meta`, `generated_at`, `render_mode`, `body_level`, plus the order field) |
| **Different orders** | Refused with `OrderMismatch` | (intentional refusal — fine) |
| **Same order, mixed atomicity** | Refused with `AtomicityMismatch` | (intentional refusal — fine) |

Source: `src/diff.rs`, especially `content_diff` (atomic path) and `structural_diff` (composed path). The composed path builds two `HashMap<&str, &str>` keyed by id, walks both to derive added/removed/version-changed — fundamentally set semantics. Order information is dropped on construction.

### v0.2 improvement A — Sequence-aware structural diff

**The user's specific request: "diffing what seed-ids/file names exist in a sequence (both additions and subtractions like traditional diffing)."**

#### Problem

Composed elements have an **ordered** `composed_of` array. The order is load-bearing — Compose emits inputs in that order, the body wraps them in that order, `--text` renders them in that order. Reordering inputs produces a different rendered prompt.

But v0.1's `structural_diff` treats `composed_of` as a set. Two compositions with the same inputs in different orders are reported as `recipes_equal: true`. This is wrong for the actual use case.

A worked example:

```toml
# Composition A: composed_of order is [role, safety, format]
composed_of = [
  { id = "role-statement",  version = "1.0.0" },
  { id = "safety-fence",    version = "1.0.0" },
  { id = "format-rules",    version = "1.0.0" }
]

# Composition B: same elements, but order is [safety, role, format]
composed_of = [
  { id = "safety-fence",    version = "1.0.0" },
  { id = "role-statement",  version = "1.0.0" },
  { id = "format-rules",    version = "1.0.0" }
]
```

The rendered bodies of A and B will have different H2-section orders. The pasted prompt the LLM sees will be ordered differently. But `oovra compare A B` today says: identical recipe.

#### Proposed behavior

Treat `composed_of` as a **list**, and use `similar::TextDiff` over a per-line representation:

```
@@ recipe (sequence diff) @@
  role-statement@1.0.0
+ safety-fence@1.0.0          <- inserted at position 2
  output-format@1.0.0
- examples-block@1.0.0        <- removed (formerly position 4)
~ tone-direct@1.0.0 -> @1.1.0 <- in-place version bump
  format-rules@1.0.0
```

This is what a working programmer expects when they say "diff." It surfaces:

- **Insertions** at specific positions
- **Removals** at specific positions  
- **Reorderings** (which show up as remove-from-old-position + insert-at-new-position pairs)
- **Version changes** as in-place "modify" events

#### Why this can't be cleanly replicated with existing operators

You could write:

```bash
oovra decompose --format=json A.md | jq -r '.inputs[] | "\(.id)@\(.version)"' > /tmp/a-recipe
oovra decompose --format=json B.md | jq -r '.inputs[] | "\(.id)@\(.version)"' > /tmp/b-recipe
diff /tmp/a-recipe /tmp/b-recipe
```

This works today and produces a valid sequence-aware diff. **It's a discovered feature** worth documenting. But it has friction:

- Requires `jq` (external dependency users may not have)
- Needs two temp files
- Loses the `compare`-specific UX (colors, JSON output for agents, the `recipes_equal` summary)
- Doesn't combine with header-field diffs or body diffs (you have to run multiple commands)

A built-in sequence mode is a small, contained addition to `compare` that consolidates the workflow.

#### Implementation sketch (for v0.2 planning, not code yet)

- Add a `--mode <set|sequence>` flag to `compare` (or expand the existing `StructuralDiff` to include both axes by default)
- Default to **sequence** mode — it's what programmers expect from "diff," and it's a superset of set information
- Internally: build `Vec<String>` of `format!("{}@{}", id, version)` for each side, run `TextDiff::from_lines`, emit unified-diff-style output
- For JSON: extend `StructuralDiff` with a `sequence_diff` field containing a structured list of `{op: Insert|Delete|Modify, position: usize, ...}` events
- Keep the existing `added`/`removed`/`version_changed` fields for backward compatibility — they remain useful for the "I just want the set difference" query, which is a real use case for library audit and dedup analysis

#### Pitfalls to watch for

- **Reordering reporting**: a pure move (`[A, B, C]` → `[B, A, C]`) shows up in LCS-diff as `delete A at position 1, insert A at position 3` (or similar) — which is *technically* correct but slightly noisy. Some diff tools have a "detect moves" mode that recognizes this. Worth considering for v0.2, not essential for v0.1.5.
- **In-place modifies**: `[A@1.0]` → `[A@1.1]` should show as a single `~ A: 1.0 -> 1.1` line, not as `delete A@1.0; insert A@1.1`. The line-based LCS will see them as distinct lines unless you pre-process to align same-id rows. The current `version_changed` field already produces this output for the set case; the sequence diff needs the same special-casing.

### v0.2 improvement B — Body content diff for composed elements

**The user's specific request: "the actual content of the prompt itself."**

#### Problem

Right now, `structural_diff` on two composed elements returns nothing about the rendered prose. Even when the recipe (`composed_of`) is identical, the bodies could differ — and v0.1's compare gives you no visibility into that. Possible sources of body divergence when recipes match:

1. **Hand-edited body** — someone modified the composed file's body manually (against spec). The recipe says "same as before" but the bytes on disk diverged. This is exactly the case re-render is designed to detect, but re-render fixes the drift instead of reporting it.
2. **Library drift without version bump** — two compositions point at `role-statement@1.0.0`, but at different times someone edited `role-statement.md` without bumping its version. The two composed files were generated at those different times.
3. **Future render-mode drift** (v0.2+) — if `render_mode` changes between compositions, the rendered bodies will look very different even with identical recipes.

#### Proposed behavior

`compare` should detect and report body divergence. Two implementation paths:

**Option B1: Render-then-diff** (using `compose --text` internally)

Build `render_for_paste`-equivalent prose for both A and B, then run `similar::TextDiff::from_lines` over the result. Output: unified-diff-style report of prose changes, plus a `bodies_diverged` boolean.

This catches the "same recipe but different prose" case at the rendered-prose level. Most informative for the user.

**Option B2: Per-element body diff (granular)**

For each input ID present in both compositions, decompose both, find that input in each, content-diff the two recovered bodies. Output: per-element body diffs.

More granular but harder to present. Useful for "show me exactly which embedded input drifted."

For v0.2, **Option B1 is the right starting point** — it gives the most user-useful output ("here's what changed in the actual prompt prose"), and it composes well with existing flags (you'd add `--with-body` to compare, or make it default).

#### Replicability check

You can do Option B1 today with:

```bash
oovra compose --library X --text $(oovra decompose --format=json a.md | jq -r '.element_id') > /tmp/a.txt
# ...and similar for b, then diff
```

But: this requires the **library** to be present, since `compose --text` resolves IDs against the library. The composed files themselves are self-describing (their bodies have everything embedded), but `compose --text` reads from the library, not from the file.

**Insight**: there's a subtle gap here. `compose --text <id>` works on library IDs, not on file paths. To render the prose of an arbitrary composed file standalone, you'd need a different code path that reads from the file's body directly. v0.1 doesn't expose that. A built-in compare-with-body would internally use the file's body (via decompose's recursive logic), making it work on standalone files with no library access — that's a meaningful capability, not just a UX win.

#### Why this isn't a duplicate of `decompose --full` + external diff

You could:

```bash
oovra decompose --full -o /tmp/a a.md
oovra decompose --full -o /tmp/b b.md
diff -r /tmp/a /tmp/b
```

This works! It would surface every byte-level difference, including recursive ones. It's a **valid discovered feature** for the deep-comparison case.

The built-in compare-with-body is different in two ways:
1. It works on **rendered prose** (the H2-flattened paste-ready form), not the raw self-describing form. The raw form has timestamps and metadata in every embedded chunk that produce noise in the diff; the rendered form has only the prose.
2. It produces a structured `BodyDiff` that integrates with the rest of the `DiffReport` JSON, suitable for agentic pipelines.

So `decompose --full + diff -r` is the right tool for "tell me everything that's different at byte level." `compare --with-body` is the right tool for "tell me what changed in the actual prompt my model would see."

### v0.2 improvement C — Broader header-field tracking

**The user's specific request: "the content of the other TOML header stuff."**

#### Problem

`content_diff` tracks `name`, `version`, `meta`. It silently ignores changes to every other field. `structural_diff` tracks nothing beyond `composed_of`.

Fields that should be tracked but currently aren't:

| Field | When it should be tracked | Significance of a change |
|---|---|---|
| `id` | both atomic and composed | A rename. Usually intentional but worth surfacing. Currently dropped entirely. |
| `order` | composed-vs-composed | Order changes indicate the input set's compositional depth changed — useful diagnostic. (Atomic-vs-atomic is always order=0, so no diff possible there.) |
| `generated_at` | composed-vs-composed | Always changes on re-render. Mostly noise. Could be summarized as a single "regenerated N hours ago" line rather than emitted as a field change. |
| `render_mode` | composed-vs-composed | Will be load-bearing in v0.2 when multiple renderers exist. A render_mode change explains a body diff that would otherwise look mysterious. |
| `body_level` | composed-vs-composed | Changes indicate the input set's max-order shifted (e.g., an order-0 input was promoted to order-1). Useful diagnostic. |

#### Proposed behavior

Extend `track_field` calls to cover all fields, with smart formatting for noise:

- `generated_at` displayed as `(regenerated ~3 hours later)` rather than the full timestamp pair, unless `--show-timestamps` is passed
- Atomic-vs-atomic: track id, name, version, meta — and surface any "atomic field present that shouldn't be" as a validation warning (this would surface validator bypasses)
- Composed-vs-composed: track id, name, version, meta, render_mode, body_level — surface order changes prominently — relegate generated_at to optional display

#### Replicability check

You could do this today with:

```bash
head -20 a.md > /tmp/a-header
head -20 b.md > /tmp/b-header
diff /tmp/a-header /tmp/b-header
```

This shows raw frontmatter differences but doesn't:
- Distinguish meaningful changes (name, version) from expected drift (generated_at)
- Produce structured field-change output for JSON consumption
- Integrate with the recipe diff and body diff in one report

Built-in version is a small extension and gives a more useful single-shot view.

### v0.2 improvement D — Recursive diff for nested compositions

#### Problem

Both `content_diff` and `structural_diff` operate at the **immediate level only**. For an order-2 composition, `structural_diff` reports changes to the order-1 inputs but doesn't tell you what changed inside those order-1 sub-prompts.

If you want to know "for this order-3 element, which atomic leaf at which depth changed," you have to manually decompose both files and recurse.

#### Proposed behavior

`compare --recursive` walks both compositions tree-style. At each level, it records:

- For atomic leaves: content-diff
- For composed elements: structural-diff (with sequence info per improvement A)

Output is a tree-shaped diff:

```
~ coding-agent (order 2)
  ~ agent-foundations (order 1)
    ~ role-statement: version 1.0.0 -> 1.1.0
      ~ body diff: -2 +3 lines
    = safety-fence: unchanged
  = output-discipline (order 1, unchanged)
```

#### Replicability check

```bash
oovra decompose --full -o /tmp/a a.md
oovra decompose --full -o /tmp/b b.md
# Then for each pair of corresponding files:
diff -r /tmp/a /tmp/b
```

`diff -r` shows you everything that's different — but it shows byte-level differences without semantic interpretation. It can't say "this version bump on a deep leaf explains the cascade of body changes you see in the parent." A recursive compare can.

So this is **partially replicable** but the built-in version produces a meaningfully richer report. Moderate-priority addition.

### v0.2 improvement E — Word-level body diff option

#### Problem

`content_diff` uses `TextDiff::from_lines` — line-level granularity. For paragraph-shaped prose this is usually right. For small inline edits ("the user" → "a user"), line-level shows the entire line as `- ... + ...`, hiding the actual word that changed.

#### Proposed behavior

`compare --word-level` (or `--granularity=word|line|char`) selects the diff granularity. `similar::TextDiff::from_words` and `from_chars` already exist; the change is selecting between them.

#### Replicability check

External tools like `git diff --word-diff` or `wdiff` handle this. Replicable but requires the user to use external tools and lose the structured-output integration. Low-priority but easy to add.

---

### v0.2 improvement F — Compare across orders / atomicity (opt-in)

#### Problem

v0.1 refuses cross-order and cross-atomicity comparisons. This is the right default — those cases are ambiguous. But there are real use cases where a user knows what they want:

- **"Does this atomic appear inside this composed element?"** — could be answered by recursively decomposing and checking. A `compare --contains <atomic-id>` flag could do this directly.
- **"Show me the prose of the order-1 input alongside the prose of its atomic-equivalent author note."**

These are arguably new operators rather than compare modes. For v0.2, leaving these as refusals (and documenting `decompose --full + grep` as the workaround for "is X in Y") is probably right.

## Part 2 — Discovered Features in v0.1

These are capabilities that already exist via combinations of existing operators, but aren't documented as features. They should be added to the reference docs so users know about them.

### D-1. `compose --text <id>` as a validation oracle

Running `oovra compose --library ./lib --text <id>` loads the element, runs the full validator, renders the body — but writes nothing. If it succeeds and produces non-empty output, the element is fully valid against the v0.1 schema.

This works for both atomic and composed elements. For atomic: validates the element itself. For composed: validates the element AND its body's recursive sub-element structure (the body parser is invoked as part of render_for_paste's recursion).

**Suggested doc placement**: `Documentation/reference/command-compose.md` already mentions this briefly. Should be promoted to its own section as "Validation oracle pattern."

### D-2. `compose --text <single-atomic-id>` as a pretty-printer

For an atomic element, `compose --text <id>` outputs exactly `## <id>\n\n<body>`. This is a "show me just the prose of this element" pretty-printer that strips TOML frontmatter. Useful for inspecting library content without `cat`-ing the raw file.

### D-3. `decompose --full` as a self-extracting archive

Hand someone an order-N composed file with no library access. They run `oovra decompose --full -o ./recovered file.md` and get every constituent atomic at byte precision. The file IS the library, packed into one document.

This means a composed file can be used as a "snapshot mechanism" for a whole prompt library: compose your atomics into an order-N umbrella element, share the umbrella file, recipients can `decompose --full` to reconstruct the library.

### D-4. `compose --re-render` as a drift detector

If `oovra compose --library X --re-render file.md` succeeds, the file's recipe is internally consistent with the library. If running it on an already-rendered file produces different bytes than what was on disk, **someone hand-edited the body** (against spec). Diffing the pre-rerender file against the post-rerender file shows you exactly what they changed.

```bash
cp file.md /tmp/before
oovra compose --library ./lib --re-render file.md
diff /tmp/before file.md
```

Suggested doc: `Documentation/reference/command-compose.md` § Mode 3 should mention this as a hand-edit-detection pattern.

### D-5. External diff over `compose --text` for rendered-prose comparison

When the user wants to diff the **rendered prose** of two composed elements (which v0.1's `compare` doesn't do — see improvement B), they can:

```bash
oovra compose --library ./lib --text a > /tmp/a.txt
oovra compose --library ./lib --text b > /tmp/b.txt
diff /tmp/a.txt /tmp/b.txt
```

This produces a clean word-or-line-level diff of just the prose. Requires the library to be present (because `--text` resolves by ID).

Variant for standalone files (no library):

```bash
# Render via decompose + manual stitching
oovra decompose --full -o /tmp/a a.md
oovra decompose --full -o /tmp/b b.md
diff -r /tmp/a /tmp/b   # byte-level recursive diff
```

The second variant works on files alone with no library, at the cost of being byte-level rather than rendered-prose-level.

### D-6. `compare --format=json | jq` for batch operations

The JSON output of `compare` is structured and stable. Pipelines:

```bash
# Find compositions where only versions changed (no add/remove)
oovra compare --format=json a.md b.md | jq 'select(.added | length == 0 and .removed | length == 0)'

# Extract just version bumps as id|old|new tuples
oovra compare --format=json a.md b.md | jq -r '.version_changed[] | "\(.id)|\(.before_version)|\(.after_version)"'
```

This is the standard "Unix-philosophy structured output" pattern, available because the diff structs derive `Serialize`. Worth highlighting in the compare docs.

### D-7. `decompose --full` + git for library forking

```bash
oovra decompose --full -o ./my-fork umbrella.md
cd ./my-fork && git init && git add -A && git commit -m "fork of umbrella v1.0.0"
```

You now have a working git repo with the entire prompt library that produced `umbrella.md`. Diverge from there. Useful for "I want to take this existing prompt and rework it."

### D-8. `compose --text` of an order-1 element as a "preview" mechanism

Before committing a composed file to disk, run `--text` to see the rendered prose. If it looks right, run `compose` (without `--text`) to write the file. The two operations produce semantically equivalent prose content; one is for inspection, the other for storage.

This is implicit in v0.1 but worth calling out as the standard authoring workflow.

## Part 3 — Missing Features in v0.1 (with replicability filter)

For each potential v0.2 feature, I've checked: **can existing operators replicate this?** If yes, the feature gets demoted to "documented as discovered" (see Part 2) unless the built-in version provides a meaningful additional capability.

### Tier A — Genuinely new, NOT replicable

#### A-1. `oovra rename <old-id> <new-id> --library <DIR>`

**Capability**: Rename an element AND update every reference to it across the library.

**Why not replicable**: A rename has two parts — rename the file itself (`mv old.md new.md`, then edit the frontmatter `id`), AND update every `composed_of` entry across the library that referenced the old ID. The second part is hard to do correctly with shell scripting because it must be aware of TOML structure (regex on `id = "old"` could accidentally match a string body that happens to contain that text). A built-in is much safer.

**Use case**: "I named this element badly when I created it; fix it without breaking the world."

#### A-2. Multi-renderer support: `render_mode = "claude-xml" | "plain-text" | "openai-md"`

**Capability**: Compose can emit prose in different prompt-engineering formats. Claude-XML wraps each section in `<role>...</role>` tags; plain-text strips all Markdown formatting; OpenAI-MD uses different conventions.

**Why not replicable**: v0.1 hardcodes `markdown-h2`. You could post-process the `--text` output with sed/awk to convert to other formats, but you'd lose the structured rendering (e.g., Claude-XML expects specific element-name conventions that a generic post-processor can't infer).

**Implementation note**: The `render_mode` field already exists in the schema. Adding new modes is purely a code addition — schema is forward-compatible.

#### A-3. `oovra rdeps <id>` (reverse dependency)

**Capability**: "Which composed elements transitively use this atomic/element?"

**Why not replicable**: Today you'd `grep -rl 'id = "target"' library/` — but that gives you false positives (any text body that mentions the ID matches) and false negatives (different formatting in `composed_of`). A built-in walks the library properly, parses each composed element's `composed_of`, and reports the actual dependents.

**Use case**: "Before I delete or rename this atomic, who would break?" Critical for library hygiene.

#### A-4. Semver range matching for version pins

**Capability**: `composed_of` entries can use `^1.0` / `>=1.2, <2.0` instead of exact versions; resolution finds the latest matching version in the library.

**Why not replicable**: Exact-string match is hardcoded in the resolver. The `semver` crate is already a dependency (it parses semver), but range matching requires `semver::VersionReq` and a slightly different resolution loop. Cannot be added externally.

**Use case**: Loose pinning during development; tight pinning at release time via `--re-render`.

#### A-5. The `bundle` kind / schema version

**Capability**: A `bundle.md` file that groups related atomics for distribution. v0.2 schema would add a `kind = "bundle"` discriminator (or extend the existing schema with a new field) and a bundle-extraction operator.

Plus: add a `schema_version = "0.2"` field to all files (atomics, composed, and bundles), with v0.1 files assumed to be schema 0.1.

**Why not replicable**: New file kind, new operator, new schema field. Can't be backported with existing tools.

**Use case**: Sharing a curated set of atomics (e.g. "coding-assistant atomics v1") as a single distributable unit.

#### A-6. Compose with inline version pin syntax: `oovra compose role@1.0.0 safety@2.0.0`

**Capability**: When composing fresh, allow pinning each input to a specific version. Currently only `--re-render` enforces pins.

**Why not replicable**: The library function `compose()` already accepts an `Option<String>` pin per input — but the CLI never exposes that field; it always passes `None`. Adding this is a CLI-layer change.

**Use case**: "Reproduce this exact composition months later, even if the library has moved on."

#### A-7. `oovra create --copy <existing-id> --new <new-id>`

**Capability**: Clone an existing element under a new ID. Useful for "fork this prompt and modify."

**Why not replicable**: You can do `cp library/old.md library/new.md && sed -i 's/id = "old"/id = "new"/' library/new.md && sed -i 's/name = "Old"/name = "New"/' library/new.md`, but it's clumsy and error-prone (the sed regexes could match body text). Built-in version uses the parser to make this safe.

**Use case**: Iterate on prompt variants without manual file gymnastics.

#### A-8. Compose dry-run with detailed report: `oovra compose --check role-statement safety-fence`

**Capability**: Resolve all inputs, run the full validator, report what WOULD be produced (output order, body_level, total byte size, leaf count) — without writing anything.

**Why partially replicable**: `compose --text <ids>` is close to this — it loads, validates, and renders without writing. But `--text` produces prose output, not a structured report about what the FILE would look like. A check flag would report on the FILE structure (order, body_level, embedded leaf count, total size) — different information.

**Decision**: This might be a `discovered feature` instead. The compose validation already happens via `--text`. The "what the file would look like" information is partial-replicable via `oovra compose <ids> -o /tmp/preview.md && head -20 /tmp/preview.md`. So borderline. Probably defer to v0.3.

### Tier B — Partially replicable, but built-in adds meaningful capability

#### B-1. `oovra audit <library>` — library hygiene report

**Capability**: A single command that runs every known integrity check:
- All elements parse?
- Any duplicate IDs?
- Any composed elements with `composed_of` pointing at missing IDs?
- Any composed elements that can't be re-rendered (version drift in library)?
- Any orphaned atomics (not used by any composition)?
- Any composed elements where `body_level` doesn't equal `max(input.order) + 1` (would indicate hand-edited corruption)?

**Why partially replicable**: Each individual check is doable. Walking the whole library and aggregating them is scriptable (with shell or jq pipelines). But the value of a single `oovra audit` is that it's a one-liner in CI, with structured output, and the report knows what to look for.

**Use case**: CI integration. Library health check. Pre-publish verification.

#### B-2. `oovra deps <id>` — forward dependency tree

**Capability**: Show the recursive dependency tree of an element. For atomic: empty. For composed: the tree.

**Why partially replicable**: `oovra decompose --full -o /tmp/x file.md && find /tmp/x` gives you everything, but as files on disk rather than as a tree report. The built-in version doesn't touch disk and produces a structured tree (JSON or human-readable).

**Use case**: Quick "what's this composed of" inspection without disk side-effects.

#### B-3. `oovra grep <pattern> --library <DIR>` — full-text search across library

**Capability**: Search element bodies (and optionally metadata) for a regex pattern.

**Why replicable but worth having**: `grep -r 'pattern' library/` works, but it doesn't distinguish frontmatter matches from body matches, doesn't show element metadata for matches, and doesn't recurse into composed-element bodies (where the embedded atomic content is wrapped in delimiters that confuse grep).

**Use case**: "Where do I talk about safety in my library?"

#### B-4. Standalone-file `compose --text` (no library required)

**Capability**: Render the prose form of a composed file without needing the library to be present.

**Why this is more than replicable**: v0.1's `compose --text` looks up elements in a library by ID. If you have a standalone composed file but no library, you can't render it. But the composed file's body has all the inputs embedded — decompose can recover them. A `--text-from-file <PATH>` flag would render the prose directly from the file's body without library lookup.

This is internally a thin wrapper over `decompose` + `render_for_paste`, but exposing it removes an obstacle.

**Use case**: Sharing composed files where the recipient doesn't have your library.

### Tier C — Replicable, document as discovered features

The following were considered but are well-served by combining existing operators. They become "discovered features" in Part 2:

- `oovra validate <path>` → replicated by `compose --text <id>` returning success/failure (D-1)
- `oovra inspect <path>` → replicated by `cat path` for atomic + `decompose --format=json path` for composed
- Library element listing → replicated by `for f in library/*.md; do head -7 "$f"; done`
- Render-prose diff between two composed files → replicated by D-5
- `oovra stats <library>` → replicated by shell pipelines on `decompose --format=json` outputs

If we ever felt these were too verbose, they could be added as thin convenience wrappers in v0.3+. They're not v0.2 priorities.

## Part 4 — Build & Distribution Improvements (Tier D)

These are not feature changes but ship/distribution improvements that significantly affect adoption.

### D-1. Publish to crates.io

`cargo publish` would make `cargo install oovra` work without needing to clone the repo. Requires:
- Choosing a name not taken on crates.io
- Producing a polished crates.io README
- Removing any private dependencies

### D-2. Pre-built binaries via GitHub Releases

For users who don't have a Rust toolchain, ship pre-compiled binaries per platform via GitHub Releases. Pair with `cargo binstall oovra` for the auto-download install path.

### D-3. CI for cross-platform test verification

GitHub Actions workflow that runs `cargo test` on Linux + macOS + Windows on every push. Verifies the cross-platform claim continuously.

### D-4. CI for documentation snapshot regeneration

A workflow that re-runs the demos in `Documentation/demos/` and verifies the captured outputs (SHA256 sums, file sizes, structural diffs) match what's committed. If they drift, fail CI and prompt regeneration. This addresses the "snapshot can go stale" risk in the docs.

## Part 6 — Architectural notes for v0.2 implementation

A few design observations that should guide v0.2 work:

### O-1. The schema is already forward-compatible

Adding new conditional fields, new render modes, or new optional fields will not break v0.1 files. The validator's joint invariant (`composed_of` presence ↔ all four companion fields) means new conditional fields can join the companion set without disrupting existing files.

### O-2. The discovered features (Part 2) should be promoted to documentation before v0.2 ships

Many "missing features" in the original plan turn out to be replicable today. If we don't document the discovered features, users will continue to perceive these as gaps and request them. Updating the reference docs to call them out explicitly is a v0.1.5 task that should happen before any v0.2 code work.

### O-3. Compare improvements naturally cluster

The sequence-aware diff, broader header tracking, and body content diff are all extensions to `DiffReport`. They can be implemented as additive variants:

```rust
pub enum DiffReport {
    Content(ContentDiff),        // current
    Structural(StructuralDiff),  // current
    Recursive(RecursiveDiff),    // NEW (v0.2)
}

pub struct ContentDiff {
    // existing fields...
    pub all_field_changes: Vec<FieldChange>,  // expanded to cover id, generated_at, etc.
    // ...
}

pub struct StructuralDiff {
    // existing fields (added/removed/version_changed) — keep for set view
    pub sequence_diff: SequenceDiff,    // NEW — positional info
    pub body_diff: Option<BodyDiff>,    // NEW — rendered prose diff
    pub all_field_changes: Vec<FieldChange>,  // NEW — header field changes
}
```

The existing fields don't have to change — they remain accurate as "set view." The new fields add the missing axes.

### O-4. The CLI flag surface for compare needs a design pass

Once compare has 3–5 distinct diff axes (sequence, set, body, header, recursive), the CLI flags need to be clean. Options:

- All-on-default with `--no-X` flags to suppress noisy axes
- Selectable axes via `--show=sequence,body,header` (positive selection)
- Smart defaults: full report in `--format=json`, summary report in `--format=human`

Probably defer this until the feature work is mostly done; pick the flag scheme that matches how the diffs actually feel in practice.

## Part 7 — What's explicitly NOT in v0.2 scope

To prevent scope creep, the following are explicitly **deferred**:

- TUI (terminal UI) for browsing the library — interesting standalone Rust project, not v0.2
- Obsidian plugin — separate ecosystem, separate maintenance burden
- IDE integration (VSCode/JetBrains extensions) — same
- A web playground — same
- Round-trip parsing of arbitrary external prompts back into atomics (the hard SPLIT direction — would require NLP-style boundary detection)
- Internationalization / non-ASCII ID support — the kebab-case grammar is intentionally ASCII-only for v0.2
- Custom user-defined render modes via plugins — over-engineering for v0.2
