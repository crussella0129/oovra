# Oovra v0.2 Specification

**Status:** Draft 3
**Author:** Charles Russell
**Supersedes:** v0.1 schema and CLI as of repo head `crussella0129/oovra@main`
**Scope:** Schema migration from numeric `order` to `kind = "atom" | "compound"`, supporting CLI changes, validator rewrite, back-compat loader, license fix, and the test surface that must change.

**Change log from Draft 2:** clarified the type taxonomy as "class **element**, kinds **atom** and **compound**" (┬¦1.5) and added ┬¦0 to anchor the audience by pointing at the primary use cases ŌĆö authoring, versioning, and analyzing system prompts and agent-instruction files (`claude.md`, `codex.md`, `AGENTS.md`, etc.) used by API-driven and local LLM systems.

**Change log from Draft 1:** the discriminator values are now `"atom"` and `"compound"` rather than `"element"` and `"compound"`. The umbrella type stays `PromptElement` (a prompt element is either an atom or a compound). See ┬¦1.5 for the rationale.

This is a migration document, not a rewrite. v0.1 is roughly 2,500 lines of Rust across `src/` and `tests/`; v0.2 changes maybe 600 of those lines and adds a few hundred more. Most of the work is renaming a field and tightening a validator. The hard part is honest accounting of what the rename gives up and what it preserves.

---

## 0 What Oovra is for (audience anchor)

Oovra is a version-controlled authoring and analysis tool for **the prose that configures LLM systems** ŌĆö system prompts, agent-instruction files, and the small policy documents that increasingly govern how AI tools behave. Concretely, it is built to solve four classes of problem:

1. **System prompts for API-driven applications.** When a team builds anything on the Anthropic API, OpenAI API, or any other LLM backend, the system prompt becomes a production artifact ŌĆö it ships with every request, it determines behavior, and changes to it are deployments. Today these are typically stored as a single string constant or a single Markdown file with no version semantics and no compositional structure. Oovra treats the system prompt as a *compound* assembled from versioned *atoms* (role, tone, refusal policy, output format, examples), each independently editable, comparable, and reusable across products.

2. **Agent-instruction files for coding agents and local tools.** Files like `claude.md`, `codex.md`, `AGENTS.md`, `.cursorrules`, `.windsurfrules`, and the equivalents shipped by every new agent framework are prompt artifacts under another name. They live in repos, get committed alongside code, drift over time, and have no diff semantics richer than line-level Git. Oovra gives them the same atom/compound structure as a system prompt: an instruction file is a compound assembled from atoms a team can share across projects ("our standard refusal posture," "our standard output format," "our standard repo-aware role declaration"). `oovra compose --text` emits the rendered file directly; `oovra decompose` recovers the recipe; `oovra compare` diffs two versions structurally.

3. **Local LLM system prompts.** Llama.cpp, Ollama, vLLM, and every local inference setup takes a system prompt as configuration. Oovra is local-first ŌĆö no network calls, no API dependencies ŌĆö and the rendered output is plain text suitable for any inference stack.

4. **Prompt-engineering practice as a discipline.** Anyone who builds with LLMs accumulates a personal corpus of "this kind of refusal works," "this tone keeps the model calibrated," "this example block reliably teaches the format I want." Today that corpus lives in scratch notes, Notion pages, or buried in product code. Oovra gives that corpus a file format, a library structure, and four operators (Create, Compose, Decompose, Compare) so the corpus is a real engineering asset rather than tribal knowledge.

The v0.2 schema migration is in service of all four. The atom/compound vocabulary is concrete enough that an `claude.md` file generated by `oovra compose` is self-documenting about what it contains and where it came from ŌĆö a property no plain-Markdown agent-instruction file currently has.

## 1 Motivation

### 1.1 What's wrong with `order`

The v0.1 schema uses a non-negative integer `order` field as the kind-discriminator. The compose-time formula in `src/render.rs::compute_order` is:

```text
let H = max(input.order for input in inputs)
let C = count of inputs where input.order == H
output_order = if C > 1 then H + 1 else H
```

This formula is **path-dependent**: the order of an artifact depends on the bracketing the user chose during one particular `compose` call, not on any intrinsic property of the resulting prompt content.

**Worked counterexample.** Take five atoms `a, b, c, d, e` (in v0.1 these were order-0 elements; under the new vocabulary they're atoms).

- **Path 1.** `compose a b c d e` ŌåÆ one compose call with five order-0 peers. Result: `order = 1`.
- **Path 2.** `compose a b` ŌåÆ `ab` (order 1). `compose c d` ŌåÆ `cd` (order 1). `compose ab cd e` ŌåÆ two order-1 peers. Result: `order = 2`.
- **Path 3.** `compose a b` ŌåÆ `ab` (order 1). `compose ab c` ŌåÆ still order 1 (only one peer at the top). `compose abc d` ŌåÆ still order 1. `compose abcd e` ŌåÆ still order 1. Result: `order = 1`.

All three artifacts contain the same five atoms in the same final concatenation order. The rendered prompt is, modulo whitespace, identical. But the on-disk `order` differs across paths.

This means `order` is not a property of the prompt ŌĆö it is a fingerprint of the user's grouping history during composition. A fingerprint is fine if it has a use; this one does not. No operator in v0.1 consumes `order` semantically. `compare` uses it only to gate atomic-vs-composed dispatch, which is `is_atomic()` in disguise. `decompose` uses it only as display metadata. `compose` writes it but never reads it back. The `order` field carries information no operator depends on.

### 1.2 What `body_level` does that `order` doesn't

`body_level`, computed by `compute_body_level` as `max(input.order) + 1`, has none of these problems. It is monotonic in nesting, it depends only on the maximum input depth (not the count at that depth), and the body parser in `decompose::decompose` depends on it directly: the parser uses `body_level` to construct the tilde-count it scans for. The chiral-delimiter collision-freeness proof (each level has strictly more tildes than any nested level) goes through `body_level`, not `order`.

**The information `order` was claiming to carry ŌĆö "compositional depth" ŌĆö is in fact already carried by `body_level - 1`.** The v0.1 SCHEMA section "Two integers, not one" tries to argue that `order` and `body_level` answer "different questions," but the question `order` answers is "how did the user happen to bracket their inputs," which is the wrong question.

### 1.3 The atom/compound discriminator

A binary `kind` field is honest about what's knowable from the artifact itself:

- **`kind = "atom"`** ŌĆö the file was hand-authored. It has no `composed_of` recipe. Its body is freeform Markdown. An atom is the smallest indivisible unit of prompting content. This is `is_atomic()` from v0.1, promoted from a computed predicate to a stored field with a sharper name.
- **`kind = "compound"`** ŌĆö the file was produced by `oovra compose`. It has a `composed_of` recipe, a `body_level`, a `generated_at`, and a `render_mode`. Its body is wrapped sub-prompt-element files. This is `is_composed()` promoted to a stored field.

A compound can contain any mixture of atoms and other compounds. The body's chiral-delimiter level still escalates monotonically with `body_level`, so the parser's collision-freeness is preserved untouched. Decompose-then-recompose with a different bracketing produces a different compound with the same atoms ŌĆö and the schema no longer lies about it.

For users who want a depth metric for filtering or sortability, v0.2 introduces an **optional, derived** `depth` field on compounds, defined as `1 + max(child.depth, default = 0)` where atoms have `depth = 0` by convention. Unlike `order`, `depth` is computed from the recipe and is intrinsic to the artifact's content tree.

### 1.4 What this is not

This migration does not:

- change the chiral delimiter scheme or the body parser
- change the lossless round-trip property of compounds
- change the `composed_of` array format
- change the four operators' high-level semantics (Create, Compose, Decompose, Compare)
- change the Obsidian compatibility of the library directory

Those properties are load-bearing for adoption. This migration is strictly a vocabulary and validator change underneath them.

### 1.5 Why "atom" and not "element" (and the class/kind taxonomy)

The clearest way to state the type system is in two layers:

> **Class:** every Oovra file is a `PromptElement`.
> **Kind:** every `PromptElement` is either an **atom** or a **compound**.

"Element" is the class ŌĆö the umbrella that everything belongs to. "Atom" and "compound" are the two kinds that the class admits. This maps cleanly onto Rust: the *class* is the struct (`PromptElement` / `PromptElementHeader`), the *kind* is the enum (`PromptElementKind::Atom` / `PromptElementKind::Compound`), and the stored discriminator `kind = "atom" | "compound"` is the wire form of that enum.

A natural first instinct (and the working name through Draft 1 of this spec) was to call the two kinds `"element"` and `"compound"`. This was rejected because it collapsed the class and one of its kinds into the same word. If `"element"` named both the umbrella and the atomic kind, then a compound would be a "prompt element" but not an "element" ŌĆö exactly the kind of seam that produces ambiguous docs, error messages, and bug reports.

The chemistry metaphor is cleaner when you use it correctly. In chemistry an *element* is a class of substance (hydrogen, oxygen, gold) whose smallest representative is an *atom*. Atoms of one or more elements combine to form *compounds*. Oovra borrows the same hierarchy:

- A **prompt element** is a class of artifact ŌĆö content meant to configure an LLM.
- An **atom** is a single, hand-authored, indivisible prompt element.
- A **compound** is a prompt element produced by composing atoms (and possibly other compounds).

Every literate adult has been trained on this hierarchy since middle school, so it lands without a glossary.

The naming map:

| Concept | v0.1 term | v0.2 term |
|---|---|---|
| The class (any Oovra file) | prompt element | **prompt element** (unchanged) |
| The Rust struct for any Oovra file | `PromptElement` | `PromptElement` (unchanged) |
| The library directory | `elements/` | `elements/` (unchanged) |
| Kind: hand-authored, indivisible | order-0 element | **atom** |
| Kind: composed, has a recipe | order-N element (N Ōēź 1) | **compound** |
| Predicate: "is this an atom?" | `is_atomic()` | `is_atom()` |
| Predicate: "is this a compound?" | `is_composed()` | `is_compound()` |

Reading rule: when you see **element** alone, it means the *class* (any Oovra file). When you see **atom** or **compound**, it means specifically that *kind*. The stored `kind` field always uses the kind-specific terms.

This trades one syllable of "atom" for the cognitive cost saved by never having to translate "element-the-kind" vs. "element-the-class" while reading the docs. Net win.

## 2 Schema changes

### 2.1 New required field on every prompt element

```toml
kind = "atom"   # or "compound"
```

`kind` is a string with exactly two legal values: `"atom"` and `"compound"`. Any other value is a validation error. The field is required on every Oovra file. There is no default.

### 2.2 Field removed

`order` is removed from the schema. A file with an `order` field present is rejected by the v0.2 validator unless the `--legacy` loader flag is passed (see ┬¦5).

### 2.3 Field added (optional, derived)

```toml
depth = 2          # optional, compounds only, derived from composed_of
```

`depth` is a non-negative integer. When present, it must equal `1 + max(child.depth, default = 0)` where child depths are looked up by recursively decomposing the body and atoms contribute `depth = 0`. Computation is identical to v0.1's `compute_body_level` minus one ŌĆö i.e., `depth = body_level - 1`.

For v0.2 the field is optional in stored files (the validator does not require it). When `oovra compose` writes a compound, it writes `depth`. When the validator parses a file with `depth` present, it checks the equation. Writers may omit `depth` to keep diffs minimal on hand-edited or generated files.

`depth` exists for one reason: humans and tools that want to filter or sort a library by compositional depth. It is not consumed by any v0.2 operator semantically.

### 2.4 Invariant rules (the new validator)

Atom invariants:

```text
kind = "atom" ŌćÆ
    composed_of, generated_at, render_mode, body_level, depth all absent
    body is freeform Markdown, non-empty after trim
```

Compound invariants:

```text
kind = "compound" ŌćÆ
    composed_of present, non-empty array of {id, version} inline tables
    generated_at present, RFC 3339
    render_mode present, currently must be "markdown-h2"
    body_level present, non-negative integer
    body_level = 1 + max(child.body_level, default = 0)
        where child.body_level = 0 for atoms
    if depth present, depth = 1 + max(child.depth, default = 0)
        where child.depth = 0 for atoms
    body is a sequence of K complete wrapped sub-prompt-element files,
        K = len(composed_of), wrapped in level-`body_level` chiral delimiters
```

Note that `body_level` for a compound containing only atoms is `1` (same as v0.1). The collision-freeness proof is unchanged because `body_level` arithmetic is unchanged.

### 2.5 Required-field table

| Field           | Atom                       | Compound                                 |
|-----------------|----------------------------|------------------------------------------|
| `name`          | required, non-empty string | required, non-empty string               |
| `kind`          | `"atom"`                   | `"compound"`                             |
| `id`            | required, kebab-case       | required, kebab-case                     |
| `version`       | required, semver           | required, semver                         |
| `meta`          | required, may be `""`      | required, may be `""`                    |
| `generated_at`  | forbidden                  | required, RFC 3339                       |
| `render_mode`   | forbidden                  | required, `"markdown-h2"` for v0.2       |
| `body_level`    | forbidden                  | required, integer Ōēź 1                    |
| `composed_of`   | forbidden                  | required, non-empty array of `{id, version}` |
| `depth`         | forbidden                  | optional, integer Ōēź 1                    |
| `order`         | forbidden (removed)        | forbidden (removed)                      |

Atom:

## 3 CLI changes

### 3.1 `oovra create`

Behavior unchanged in spirit: scaffolds or labels an atom. Two surface changes:

1. Writes `kind = "atom"` in the frontmatter. Does not write `order`.
2. No new kind-selection flag is needed because `create` only produces atoms. Compounds are produced by `compose`.

`--new` and `--label` semantics unchanged. `--force` semantics unchanged.

### 3.2 `oovra compose`

Behavior changes:

1. Writes `kind = "compound"` in the output frontmatter.
2. Writes `body_level` exactly as v0.1 computed it (`max(input.body_level, default = 0) + 1`, treating atoms as `body_level = 0`).
3. Writes `depth` as `1 + max(child.depth, default = 0)`, treating atoms as `depth = 0`.
4. Does not write `order`.

The `--text` mode is unchanged: it still renders a flat prose form by recursively descending compounds and emitting H2 headers for each leaf atom. Note that `render_for_paste` in `src/render.rs` already does this correctly via `is_atomic()` ŌĆö no logic change needed, only a rename of the predicate to `is_atom()`.

The `--re-render` mode is unchanged in semantics. The re-rendered file inherits its own `id`, `version`, and `name` but recomputes `body_level`, `depth`, `generated_at`, and the body from the current library state.

### 3.3 `oovra decompose`

Behavior changes:

1. `decompose` (one-level) and `decompose --full` still operate on compounds and refuse to operate on atoms (with `OovraError::CannotDecomposeAtomic` renamed to `CannotDecomposeAtom`).
2. The human-formatted report changes: instead of "order N, K immediate input(s)" it prints "compound at depth D, K immediate input(s)." The JSON report's `element_order` field is renamed to `depth` and is computed from the recipe.

### 3.4 `oovra compare`

Behavior changes:

1. Dispatch is now `(a.kind, b.kind)` instead of `(a.order, b.order)`.
2. `OrderMismatch` error is renamed `KindMismatch` and its message is "Cannot compare an atom with a compound: 'a' is an atom, 'b' is a compound."
3. The `AtomicityMismatch` variant in v0.1 was already doing this work. In v0.2 it disappears, merged into `KindMismatch`.
4. The structural diff over `composed_of` is unchanged in v0.2. The compare-rewrite (Histogram diff + move detection) is **deferred to v0.3** to keep this migration tight. v0.2 ships the schema change cleanly first.

### 3.5 New: `oovra migrate`

Subcommand for one-shot migration of a v0.1 library to v0.2:

```text
oovra migrate <library-dir>
```

Behavior:

1. Walk the library directory recursively, find every `.md` file with an Oovra `+++` opener.
2. For each, parse with the legacy v0.1 schema (see ┬¦5).
3. Rewrite with the v0.2 schema: emit `kind` based on whether `composed_of` is present, drop `order`, emit `depth` for compounds.
4. Validate the rewritten file with the v0.2 validator before writing. If validation fails, abort the migration for that file with an error and leave the original in place.
5. Print a summary of migrated/skipped/failed counts.

The migration is in-place and overwrites files. The migration spec recommends running it in a clean Git working directory so the diff is auditable.

### 3.6 Flag summary

No flags are removed. New flags:

- `oovra <any> --legacy` ŌĆö accept v0.1 schema files (with `order` instead of `kind`) for read-only operations. Useful during the transition period. Errors on attempts to write legacy-format files.

(A `--no-depth` flag was considered and rejected; see ┬¦10.4.)

## 4 Validator rewrite (`src/element.rs::validate_header`)

The current v0.1 validator runs roughly 120 lines in `validate_header`. The v0.2 version is shorter because the kind discriminator is explicit.

### 4.1 New flow

```text
validate_header(header, body, path):
    require name non-empty
    require id matches kebab-case
    require version parses as semver
    # `meta` is allowed to be empty string but must be present

    require body non-empty after trim

    match header.kind:
        "atom":
            forbid composed_of, generated_at, render_mode, body_level, depth
        "compound":
            require composed_of present, non-empty
            require generated_at parses as RFC 3339
            require render_mode = "markdown-h2"
            require body_level >= 1
            for each input in composed_of:
                require input.id matches kebab-case
                require input.version parses as semver
            if depth present:
                require depth >= 1
                # the equation depth = 1 + max(child.depth) is checked
                # only by `validate-deep`, not the per-file validator,
                # because verifying it requires resolving the library
        other:
            reject with InvalidKind{ field: "kind", value, ... }
```

### 4.2 New error variants

In `src/error.rs`:

```rust
#[error("Field 'kind' in {path} has invalid value '{value}'. Expected \"atom\" or \"compound\".")]
InvalidKind { path: PathBuf, value: String },

#[error("Atom '{id}' in {path} has forbidden field '{field}'. Atoms have no recipe and no composition metadata.")]
AtomHasForbiddenField { path: PathBuf, id: String, field: &'static str },

#[error("Compound '{id}' in {path} is missing required field '{field}'.")]
CompoundMissingField { path: PathBuf, id: String, field: &'static str },

#[error("Cannot compare an atom with a compound: '{a_id}' is an {a_kind}, '{b_id}' is a {b_kind}.")]
KindMismatch { a_id: String, a_kind: &'static str, b_id: String, b_kind: &'static str },

#[error("Cannot decompose atom '{id}'. Atoms have no recipe. Only compounds can be decomposed.")]
CannotDecomposeAtom { id: String },
```

The variants `OrderRequiresField`, `OrderMismatch`, `AtomicityMismatch`, `HandAuthoredHigherOrder`, and `CannotDecomposeAtomic` are removed. Their call sites are updated to the new variants. Note that `CannotDecomposeAtom` is the descendant of v0.1's `CannotDecomposeAtomic` ŌĆö same intent, sharper noun.

### 4.3 Header struct changes (`src/header.rs`)

```rust
#[derive(Debug, Clone, Serialize, Deserialize)]
pub struct PromptElementHeader {
    pub name: String,
    pub kind: PromptElementKind,     // was: pub order: u32
    pub id: String,
    pub version: String,
    #[serde(default)]
    pub meta: String,

    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub generated_at: Option<String>,

    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub render_mode: Option<String>,

    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub body_level: Option<u32>,

    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub depth: Option<u32>,          // new

    #[serde(default, skip_serializing_if = "Option::is_none")]
    pub composed_of: Option<Vec<InputRef>>,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, Serialize, Deserialize)]
#[serde(rename_all = "lowercase")]
pub enum PromptElementKind {
    Atom,
    Compound,
}

impl PromptElementHeader {
    pub fn is_atom(&self) -> bool {
        matches!(self.kind, PromptElementKind::Atom)
    }
    pub fn is_compound(&self) -> bool {
        matches!(self.kind, PromptElementKind::Compound)
    }
}
```

The methods `is_atomic()` and `is_composed()` are renamed `is_atom()` and `is_compound()`. Every call site in `src/render.rs`, `src/decompose.rs`, `src/diff.rs`, and `src/main.rs` updates with `s/is_atomic/is_atom/g` and `s/is_composed/is_compound/g`. The Rust compiler will catch any miss.

### 4.4 Compose changes (`src/render.rs`)

`compute_order` is **removed**. The function and its tests are deleted.

`compute_body_level` is kept exactly as is. Its semantics did not depend on `order`.

A new function `compute_depth` is added:

```rust
/// Compute the recipe depth of a compound from the depths of its inputs.
/// Atoms contribute depth 0; compounds contribute their stored depth.
pub fn compute_depth(input_depths: &[u32]) -> u32 {
    input_depths.iter().copied().max().map(|m| m + 1).unwrap_or(1)
}
```

The compose function writes `kind = Compound`, omits `order` (which no longer exists), writes `body_level` and `depth`.

## 5 Back-compat: the legacy loader

A v0.1 library on disk has `order` instead of `kind`. The legacy loader translates on read.

### 5.1 Mapping rules

```text
v0.1 file with order = 0, no composed_of    ŌćÆ v0.2 kind = "atom"
v0.1 file with order Ōēź 1, composed_of set   ŌćÆ v0.2 kind = "compound"
v0.1 file with order Ōēź 1, no composed_of    ŌćÆ rejected in v0.1, still rejected
v0.1 file with order = 0, composed_of set   ŌćÆ v0.2 kind = "compound"
    (this is the "single-input edge case" from v0.1 ┬¦3.1)
```

The depth field is derived: `depth = body_level - 1` when `body_level` is present (and at least 1), else absent. This keeps v0.2's depth semantics consistent ŌĆö an atom has `depth = 0` (unstored), and a compound built from atoms has `body_level = 1` and `depth = 1`.

### 5.2 Implementation

Add a second deserialization struct in `src/header.rs`:

```rust
#[derive(Deserialize)]
struct LegacyHeader {
    name: String,
    order: u32,
    id: String,
    version: String,
    #[serde(default)]
    meta: String,
    #[serde(default)]
    generated_at: Option<String>,
    #[serde(default)]
    render_mode: Option<String>,
    #[serde(default)]
    body_level: Option<u32>,
    #[serde(default)]
    composed_of: Option<Vec<InputRef>>,
}

impl LegacyHeader {
    fn into_v2(self) -> PromptElementHeader {
        let kind = if self.composed_of.is_some() {
            PromptElementKind::Compound
        } else {
            PromptElementKind::Atom
        };
        let depth = self.body_level.map(|b| b.saturating_sub(1).max(1));
        PromptElementHeader {
            name: self.name,
            kind,
            id: self.id,
            version: self.version,
            meta: self.meta,
            generated_at: self.generated_at,
            render_mode: self.render_mode,
            body_level: self.body_level,
            depth,
            composed_of: self.composed_of,
        }
    }
}
```

The `parse` function in `src/element.rs` tries the v0.2 deserialization first. On failure with a "missing field `kind`" TOML error, it retries with `LegacyHeader` if the `--legacy` flag is set on the current operation, otherwise it returns the original v0.2 error pointing at the missing `kind` field with a suggestion to run `oovra migrate`.

### 5.3 The `--legacy` flag

The flag is global (`oovra --legacy <command> ...`). When set, parsing accepts both v0.1 and v0.2 files. Writes are always in v0.2 format. The flag does not enable writing v0.1 files. The flag exists for read-only ergonomics during the transition period (running `oovra compare` against a checked-in v0.1 file without first migrating).

The flag is removed in v0.3.

## 6 The license fix

The repo has a three-way mismatch:

- `Cargo.toml` line 8: `license = "MIT OR Apache-2.0"`
- `README.md` end-of-file: "Dual MIT / Apache-2.0 ŌĆö see LICENSE."
- `LICENSE` file contents: GNU General Public License v3, 29 June 2007.

These cannot all be true. Pick one. The Rust ecosystem norm is dual MIT/Apache-2.0; this is what `cargo new` defaults to and what almost every published crate uses. GPL-3.0 would scare off downstream commercial users (a real concern for a tool that wants adoption by prompt engineers at companies) and would make embedding Oovra into a closed-source product impossible.

**Recommendation:** keep `Cargo.toml`, replace the `LICENSE` file. Add `LICENSE-MIT` and `LICENSE-APACHE` files at the repo root, each containing the canonical text. Update `README.md` to point at both.

This is a 10-minute fix that unblocks adoption. It should land in v0.2 because the schema change is a natural breakpoint for republishing.

## 10 Open questions for the implementer

These are decisions left for the coding session, not the spec:

1. **Should `kind` come second or last in the canonical TOML field order?** Field order in TOML is preserved on serialize. The v0.1 convention puts `order` second. Putting `kind` second is consistent. Putting `kind` first (before `name`) is more discoverable for agents reading the file top-to-bottom. Spec recommendation: second, after `name`.

2. **Should the migrate command preserve `generated_at` from the v0.1 file or stamp a new timestamp?** Preserving is more honest (the artifact was generated then, not now). Re-stamping is consistent with "this is a new file." Spec recommendation: preserve, on the principle that migration is a representation change not a regeneration.

3. **Should `depth` be required or optional in the canonical v0.2 format?** Required is cleaner; optional is more forgiving for hand-edits. Spec recommendation: optional in the schema, always written by the compose tool. Validator checks the equation only when the field is present.

4. **Should the `--no-depth` flag exist?** It saves three lines per file but adds complexity. Spec recommendation: drop it. Always write depth. If users want minimal frontmatter they can hand-edit. The flag would create two valid serializations of the same artifact, which complicates future hashing work.

5. **Should `PromptElementKind` be a Rust enum or a string?** Enum is type-safe; string is more flexible if future versions add a third kind. Spec recommendation: enum. If a third kind is ever added, that's a major version bump anyway.
